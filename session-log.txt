SOCKET MODE RELIABILITY — RESEARCH NOTES (Feb 23, Session 8)
=============================================================

Root cause: Slack round-robins events across ALL WebSocket connections (live + dead).
When PM2 kills Jin without clean WebSocket close frame, stale connections persist.
No Slack API exists to list or revoke individual Socket Mode connections.

Fixes implemented this session:
  1. ecosystem.config.js — kill_timeout=10s (was PM2 default 1.6s!)
  2. Shutdown timeout guard — 8s force-exit before PM2 escalates to SIGKILL
  3. num_connections monitoring — intercepts raw WS hello message, logs count
  4. Event delivery watchdog — checks every 5min, force-reconnects after 30min silence
  5. Watchdog timer reset — on every message event and app_mention

Emergency recovery (if Jin stops receiving messages):
  Option A: pm2 restart jin (should work now with proper graceful shutdown)
  Option B: Regenerate SLACK_APP_TOKEN in Slack dashboard → paste in .env → restart
            (nuclear option: instantly kills ALL stale connections)

Long-term fix: Switch to HTTP mode using Tailscale Funnel
  - Stateless, built-in retries, no stale connection problem
  - Slack recommends HTTP for production
